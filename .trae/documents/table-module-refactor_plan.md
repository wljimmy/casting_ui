# 表格模块标准化重构计划（单文件版）

## 项目现状分析

### 当前代码状态
- `table.js` 已整合注册表、数据层、渲染层、初始化模块为单文件（约1160行）
- 测试页面使用远程JSON数据源，包含22个字段的完整测试数据
- 框架已具备基础的dom-observer、core等模块

### 存在的问题
1. **渲染层越界**：`_getDataLayer()` 返回全局对象，不符合分层架构要求
2. **容错降级不完整**：初始化失败时未完全保留原始DOM结构
3. **热更新策略不明确**：局部更新和全量重建的触发条件未严格区分
4. **测试页面冗余**：包含大量手动JS代码，不符合"框架自动初始化"原则
5. **缺少独立测试**：未按模块拆分进行独立功能测试

## 重构方案

### 目标架构（严格遵循开发文档）

```
┌─────────────────────────────────────────────────────┐
│                  初始化模块 (TableInit)             │
│  扫描 → 识别 → 注册 → 四状态机管控 → 容错降级        │
└─────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│                  注册表 (CUITableRegistry)           │
│  四状态机: pending/loading/success/error            │
│  完整字段: header, rawData, processedData,          │
│           filteredData, filterRules, sortRules,     │
│           searchRules, pageState                    │
└─────────────────────────────────────────────────────┘
                          │
        ┌─────────────────┴─────────────────┐
        ▼                                   ▼
┌─────────────────────┐         ┌─────────────────────┐
│    数据层模块        │         │    渲染层模块        │
│  (TableDataLayer)   │         │  (TableRenderLayer) │
│                     │         │                     │
│  ✅ 数据清洗/标准化  │         │  ✅ 只读注册表       │
│  ✅ 排序/筛选/搜索   │         │  ✅ 局部热更新(500ms)│
│  ✅ 分页裁切         │         │  ✅ 全量重建         │
│  ✅ 5000行阈值拦截   │         │  ✅ 防抖合并         │
│  ✅ 规则上限10条     │         │                     │
└─────────────────────┘         └─────────────────────┘
```

## 实施步骤

### 阶段1：注册表结构升级

**目标**：确保注册表结构完全符合开发文档规范

**修改内容**：
- 确认 `searchRules` 字段完整（keyword, mode, field）
- 确认 `pageState` 字段完整（pageNum, pageSize, total, pageCount）
- 完善事件通知机制，确保所有变更触发正确事件

### 阶段2：数据层模块重构（重点）

**目标**：纯数据层，零DOM操作，严格遵循阈值限制

**修改内容**：
1. 数据清洗与标准化
   - 支持多格式数据（JSON对象数组、CSV解析结果、DOM提取数据）
   - 表头适配：用户设定表头为准，数据源未找到对应列隐藏，缺少列自动空值补全
   - 纯数据按顺序填充列

2. 5000行数据量阈值拦截
   - 超过5000行直接截断，控制台警告

3. 规则堆叠逻辑
   - 筛选规则最大10条，超量拦截
   - 排序规则最大10条，超量拦截
   - 每次规则变更基于processedData全量重算

4. 分页模块
   - 纯数据裁切，不参与筛选排序
   - 参数容错兜底

### 阶段3：渲染层模块重构（重点）

**目标**：只读注册表，局部热更新与全量重建严格区分

**修改内容**：
1. 局部热更新（仅替换tbody）
   - 触发条件：表头不变、数据源结构不变
   - 防抖500ms合并更新
   - 保留冻结状态、UI状态、滚动位置

2. 全量重建
   - 触发条件：表头变更、数据源结构变更、用户新推送数据与原表头不匹配
   - 销毁重建整体表格
   - 清空历史规则，重置分页

3. 渲染层零数据修改
   - 所有交互触发调用数据层API
   - 不直接操作注册表

### 阶段4：初始化模块重构（重点）

**目标**：系统唯一入口，双向跳过判定，容错降级

**修改内容**：
1. 节点扫描跳过规则
   - DOM判定：`data-table-init=finish` 永久跳过
   - 注册表判定：`initStatus=error` 运行跳过

2. 四状态机管控
   - pending → loading（加锁，禁止重复初始化）
   - loading → success（解锁，开启全量功能）
   - loading → error（保留原始DOM，仅禁用高级功能）

3. 容错降级
   - 初始化失败：不操作DOM，不新增标记，保留原始Table结构
   - CSS样式自动生效，仅高级功能失效
   - 控制台输出错误日志

4. 数据源解析
   - 远程数据源：仅支持JSON、CSV格式
   - 本地数据源：从DOM提取
   - 非法格式直接判定初始化失败

### 阶段5：整合入口

**目标**：统一对外API，符合框架注册规范

**修改内容**：
- `CUI.Table.updateData(tableId, newOriginData)` - 用户数据更新入口
- `CUI.Table.getData(tableId)` - 获取表格数据
- `CUI.Table.filter(tableId, rule)` - 添加筛选规则
- `CUI.Table.sort(tableId, rule)` - 添加排序规则
- `CUI.Table.search(tableId, keyword, field, mode)` - 搜索
- `CUI.Table.setPage(tableId, pageNum)` - 切换页码

### 阶段6：测试页面更新

**目标**：移除冗余JS，依赖框架自动初始化

**修改内容**：
1. 简化HTML结构，仅保留原生Table标签和data-*属性配置
2. 移除手动JS渲染代码
3. 功能型表格通过`data-cui-table='{"type":"functional"}'`启用
4. 测试数据通过`data-data-source`指定远程JSON文件

### 阶段7：独立功能测试

**目标**：验证各模块功能独立运行

**测试内容**：
1. 注册表模块测试
2. 数据层模块测试
3. 初始化模块测试
