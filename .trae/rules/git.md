Git 提交规范规则（完整版）

1. 版本号统一管理
每次提交前，需按技术文档约定的规则，对全项目执行版本号更新；版本号升级需严格匹配本次提交修改的模块，禁止无关联升级或漏更版本号。

2. 配套文档同步更新
本次提交若涉及功能修改、逻辑变更、接口调整、参数变动、使用流程更新等，必须同步更新对应配套文件，包含且不限于：

• 技术开发文档

• API 接口文档（接口定义、入参出参、请求方式、状态码等）

• 产品使用手册/产品说明文档

• README 项目说明文件

• 接口注释、代码注释、模块说明文档

• 其他项目内说明性、指导性文档
禁止只提交代码、不同步更新文档的行为。

3. 提交信息标准化
提交日志需清晰、规范，格式统一，需包含：本次修改类型（feat/fix/docs/refactor等）+ 修改模块+核心修改内容，禁止使用无意义、模糊的提交描述。

4. 提交前完整性自检
提交前需自行核对：代码修改无误、版本号更新正确、所有关联文档同步更新完成、无遗漏文件、代码无明显语法/格式问题，确认无误后再执行提交。

5. 提交后记忆优化归档
每次提交完成后，及时整理本次提交的修改要点、模块变更、文档更新内容，优化记忆储存逻辑，保证记录清晰、可追溯。

6. 文档与代码一致性强制要求
所有说明性文档内容必须与当前提交的代码逻辑、功能、接口完全一致，严禁出现文档与代码脱节、描述不符的情况；若无需更新文档，需在提交信息中注明「无文档变更」。
