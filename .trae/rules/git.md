Git 提交规范规则（完整版）

## 1. 版本号统一管理
- 每次提交前，需检查所有更新元素的版本号管理情况
- 如果有更新的模块，需要同步更新版本号
- 版本号升级需严格匹配本次提交修改的模块，禁止无关联升级或漏更版本号
- 项目根目录的版本号需与所有模块版本号保持一致

## 2. 配套文档同步更新
本次提交若涉及功能修改、逻辑变更、接口调整、参数变动、使用流程更新等，必须同步更新对应配套文件，包含且不限于：
- 技术开发文档
- API 接口文档（接口定义、入参出参、请求方式、状态码等）
- 产品使用手册/产品说明文档
- README 项目说明文件
- 接口注释、代码注释、模块说明文档
- 其他项目内说明性、指导性文档
**禁止只提交代码、不同步更新文档的行为**

## 3. 版本号与文档一致性检查
每次Git提交前，必须检查以下一致性：
- **模块版本号**：检查 `src/modules/js/` 和 `src/modules/css/` 下所有模块文件的版本号
- **API文档**：检查 `docs/api/` 下的API文档中的版本号是否与模块一致
- **设计文档**：检查 `docs/` 下的设计文档中的版本号是否一致
- **README**：检查项目根目录README中的版本号是否与模块一致
- **版本日志**：检查 `docs/logs/version.md` 是否需要添加新版本记录
- **规则文档**：检查 `.trae/rules/` 下的规则文档是否需要更新

如果发现不一致，必须先更新后再提交。

## 4. 提交信息标准化
提交日志需清晰、规范，格式统一，需包含：本次修改类型（feat/fix/docs/refactor等）+ 修改模块+核心修改内容，禁止使用无意义、模糊的提交描述。

## 5. 提交前完整性自检
提交前需自行核对：
- [ ] 代码修改无误
- [ ] 版本号更新正确（所有更新的模块都已更新版本号）
- [ ] 所有关联文档同步更新完成
- [ ] 版本号与文档一致性检查通过
- [ ] 无遗漏文件
- [ ] 代码无明显语法/格式问题

确认无误后再执行提交。

## 6. 提交后记忆优化归档
每次提交完成后，及时整理本次提交的修改要点、模块变更、文档更新内容，优化记忆储存逻辑，保证记录清晰、可追溯。

## 7. 文档与代码一致性强制要求
所有说明性文档内容必须与当前提交的代码逻辑、功能、接口完全一致，严禁出现文档与代码脱节、描述不符的情况；若无需更新文档，需在提交信息中注明「无文档变更」。

## 8. 项目版本号管理
- 项目根目录的 `package.json` 中的版本号是项目整体版本号
- 每次正式提交前，需确保项目整体版本号与本次提交的最主要模块版本号一致
- 如果本次提交涉及多个模块，以最高版本号为准
