一、整体结构规范

1. 整个菜单使用\*\*<menu> 标签\*\*作为根容器，作为菜单唯一外层包裹标签
2. 菜单类型通过 <menu> 标签的类名区分，类名即菜单种类，框架通过该类名渲染对应样式与交互：

◦ 侧边栏菜单：class="menu-sidebar"

◦ 弹出式菜单：class="menu-popup"

◦ 页面内容内菜单：class="menu-inline"

1. 菜单内部仅使用\*\*<ul> + <li> 原生列表嵌套\*\*实现层级结构，禁止使用其他标签搭建菜单
2. 嵌套层级限制：最多支持 4 层嵌套，可实现「一级主菜单 + 三级子菜单」的完整层级

二、菜单项（<li>）data 属性规则
每个 <li> 为独立菜单条目，通过 data-\* 自定义属性定义功能，属性值均为文本/JSON 字符串，无需额外处理：

1. data-icon

◦ 用途：控制 <li> 菜单项前的图标展示

◦ 规则：属性值为框架内置标准图标名称，填写图标名则自动渲染对应图标，无该属性则不显示图标

1. data-badge

◦ 用途：控制 <li> 菜单项右侧的提示徽标（New / Hot / Update 等）

◦ 规则：属性值为徽标名称，填写则自动渲染对应徽标，无该属性则不显示徽标

1. data-action

◦ 用途：定义 <li> 菜单项点击后的核心跳转/加载行为，属性值为标准 JSON 字符串

◦ JSON 固定包含 3 个字段：

◦ url：需要加载的页面地址

◦ container：页面内容要刷新渲染的目标容器选择器

◦ callback：页面加载完成后立即执行的回调函数名

三、动画实现规范（核心补充）

1. 所有菜单动画统一通过 CSS 类名切换实现，JS 仅负责为 <menu>、<ul>、<li> 等菜单相关元素执行 classList.add / remove / toggle 操作，绝不直接操作元素内联样式，严格遵循样式与逻辑分离原则
2. 动画类划分与使用规则：

◦ 基础全局动画类：menu-animate（启用基础过渡）、menu-animate-fade（淡入淡出）、menu-animate-slide（滑入滑出）

◦ 子菜单层级动画类：menu-open（子菜单展开）、menu-close（子菜单折叠）、menu-collapsed（菜单整体收起）

◦ 交互状态动画类：menu-hover（悬停动效）、menu-active（选中高亮）、menu-disabled（禁用状态）

◦ 图标/徽标动画类：badge-pulse（徽标呼吸闪烁）、icon-rotate（图标旋转）

1. 动画过渡效果统一在框架 Casting.css 中定义，使用 CSS 变量控制过渡时长、曲线，沿用框架主色，符合 View Transitions API 规范
2. 多层级子菜单 <ul> 展开/折叠、菜单切换、图标徽标交互等所有动效，均通过对应动画类的增删实现，无JS直接操控样式的逻辑

四、多菜单实例化与垃圾回收规范（新增）

1. 单页面允许存在多个独立 <menu> 标签，每一个 <menu> 元素都会被自动初始化为独立菜单实例对象，实例之间数据、状态、事件完全隔离，避免多个菜单交互冲突
2. 菜单模块会自动监听页面（动态监听，要考虑到fetch过来的子页面的菜单），一旦检测到存在 <menu> 标签元素，立刻对其进行初始化并生成实例，无需手动调用初始化方法
3. 每个菜单实例单独管理自身内部 <ul>、<li> 的事件绑定、展开折叠状态、动画标识，互不干扰、不共享全局变量
4. 当包含 <menu> 标签的子页面被替换、移除时，该菜单对应的实例对象必须自动执行清理销毁：

◦ 移除该菜单所有 <li> 上绑定的事件监听器

◦ 清空实例内部存储的DOM引用、状态数据

◦ 注销实例，解除DOM与对象的关联

◦ 彻底释放内存，杜绝内存泄漏与无效监听残留

2、js操作逻辑

菜单模块完整执行逻辑规范（最终版）

1. DOM 监听与菜单识别
系统全局监听 DOM 元素，自动扫描页面中所有菜单元素。判断元素是否为框架规定的标准菜单类，若是则进入系统菜单构建流程；若为用户自定义菜单，则直接忽略，不做任何处理。

2. 配置参数提取与 JSON 化
对识别出的标准菜单元素，提取其所有属性、结构与配置数据，统一序列化为标准 JSON 对象，完成参数化处理，为后续构建提供统一数据格式。

3. 支持双模式构建菜单
框架支持两种菜单创建方式，最终共用同一套构建逻辑，保证表现与行为一致：

• HTML 预定义模式：在页面中编写基础 HTML 结构节点，由框架自动扫描、解析并完成菜单构建；

• JS API 调用模式：框架对外暴露标准化 API 接口，其他开发者可直接调用该接口，传入规范 JSON 参数，由菜单模块动态创建并渲染菜单。

4. 菜单构建与渲染
将 JSON 配置对象传入菜单构建模块，根据容器 ID、结构规则、布局规范等参数，自动生成完整菜单结构并渲染至目标位置。
构建完成后，在菜单节点上标记已构建完成状态，避免重复解析与重复构建。

5. 结构、样式、行为严格分离

• HTML 只负责提供基础结构节点与配置数据，不内嵌样式，不绑定交互行为；

• CSS 独立封装为菜单专用 CSS 模块，单独抽离为独立 CSS 文件，统一管理所有菜单样式（颜色、尺寸、布局、状态、hover/active 等），方便单独调试、修改与维护；

• JS 只负责逻辑功能与事件行为，不生成任何内联样式，仅通过增删 CSS 类实现状态切换，彻底做到样式与逻辑分离。

6. 菜单事件监听与内容加载
由系统构建完成的菜单统一由框架接管交互逻辑。JS 监听菜单点击事件，通过 fetch 异步加载对应子页面内容，并在指定容器内局部刷新展示，实现无刷新单页应用体验。

7. DOM 销毁与垃圾回收
系统持续监听菜单 DOM 生命周期，若菜单元素被删除或移出页面，自动清理对应实例、事件监听与内存引用，及时回收垃圾数据，避免内存泄漏。
