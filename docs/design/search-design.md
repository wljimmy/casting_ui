# 轻量级 UI 框架 · 通用搜索组件 API 设计

## 一、组件核心定位

**搜索组件（Search）** 是框架级**统一触发入口 + 指令分发器**，不直接操作业务 DOM、不侵入组件内部逻辑。
核心职责：

1. 统一收集用户输入关键词；
2. 按配置模式，分发指令至目标组件/自身逻辑；
3. 管理搜索全流程状态（加载、结果、空状态）；
4. 提供统一重置、结果导航能力。

## 二、核心设计原则（框架底层逻辑）

1. **解耦原则**：搜索组件不碰业务组件 DOM，仅通过 API 交互；
2. **分层原则**：公共逻辑（输入、状态、触发）与业务逻辑（DOM 筛选、文本高亮、远程请求）完全隔离；
3. **可扩展原则**：新增搜索场景仅需扩展对应模式，无需重构组件；
4. **轻量原则**：不内置历史记录、联想建议等重型功能，仅保留核心能力。

## 三、API 完整设计

### 1. 入参配置（Props）

| 参数名                | 类型     | 必选     | 默认值                   | 说明                           | 取值范围/逻辑说明                                         |
| ------------------ | ------ | ------ | --------------------- | ---------------------------- | ------------------------------------------------- |
| `mode`             | 枚举     | 是      | -                     | 搜索模式，决定指令分发方向                | `dom`（组件筛选）、`text`（文本高亮）、`remote`（远程请求）           |

### 2. 对外事件（Emits）

| 事件名       | 入参格式                                                             | 说明       | 触发场景                 |
| --------- | ---------------------------------------------------------------- | -------- | -------------------- |
| `search`  | `{ keyword: string, mode: 枚举, target?: 引用 }`                     | 搜索指令触发事件 | 点击搜索按钮、回车、输入防抖触发时    |
| `result`  | `{ keyword: string, total: number, list: 数组, currentIndex: 数字 }` | 搜索结果返回事件 | 任意模式完成匹配/请求成功后       |
| `empty`   | `{ keyword: string }`                                            | 无匹配结果事件  | 任意模式无结果时             |
| `loading` | `{ isLoading: 布尔 }`                                              | 加载状态变更事件 | 远程请求中、DOM 筛选中（轻量加载态） |
| `reset`   | -                                                                | 重置完成事件   | 调用重置方法、清空关键词后        |
| `error`   | `{ keyword: string, message: 字符串 }`                              | 错误事件     | 远程请求失败、模式配置错误时       |

### 3. 组件实例方法（Expose）

| 方法名              | 入参                                       | 返回值                                    | 说明                    | 适用模式            |
| ---------------- | ---------------------------------------- | -------------------------------------- | --------------------- | --------------- |
| `triggerSearch`  | `keyword?: 字符串`                          | 无                                      | 手动触发搜索，不传关键词则使用输入框内容  | 全模式             |

### 4. 核心状态（State）

| 状态名                  | 类型  | 说明                    |
| -------------------- | --- | --------------------- |
| `currentKeyword`     | 字符串 | 当前输入的搜索关键词            |
| `isLoading`          | 布尔  | 搜索加载中状态（远程请求、DOM 筛选时） |
| `resultTotal`        | 数字  | 本次搜索结果总数              |
| `currentResultIndex` | 数字  | 当前选中/定位的结果索引          |
| `isResetting`        | 布尔  | 重置操作中状态（避免重复重置）       |

## 四、各模式完整逻辑流转（核心业务逻辑）

### 模式 1：DOM 组件筛选（mode="dom"）

**核心逻辑**：搜索组件不操作 DOM，由目标组件自主实现筛选，仅通过 API 交互。

1. **注册校验**：使用 `registerFilter` 方法，目标组件向搜索组件注册自身筛选 API（`filterFn` 负责匹配，`resetFn` 负责恢复）；
2. **指令分发**：搜索组件触发 `search` 事件，携带关键词、目标组件引用；
3. **组件内部处理**：
   - 目标组件接收关键词，执行匹配逻辑（如菜单匹配文字、展开父级节点）；
   - 隐藏不匹配项，显示匹配项；
   - 执行 `resetFn` 时，恢复组件原始折叠/显隐状态；
4. **结果回调**：目标组件完成筛选后，调用搜索组件的 `result`/`empty` 事件回传结果；
5. **状态管理**：搜索组件同步更新 `resultTotal`、`isLoading` 状态。

### 模式 2：文本高亮（mode="text"）

**核心逻辑**：搜索组件自主处理静态文本匹配，无业务组件依赖。

1. **触发搜索**：输入关键词后，按配置执行匹配（忽略大小写、匹配规则）；
2. **文本处理**：
   - 定位目标文本容器（如富文本、文章区域）；
   - 提取文本节点，匹配关键词并包裹高亮标签（样式由框架提供类名，不内置样式）；
   - 支持清除高亮（重置时自动执行）；
3. **结果导航**：滚动定位首个匹配文本，支持上下切换结果；
4. **状态回调**：根据匹配结果触发 `result`/`empty` 事件，更新 `resultTotal`。

### 模式 3：远程请求（mode="remote"）

**核心逻辑**：搜索组件负责请求分发，结果由外部业务处理。

1. **请求校验**：检查 `requestFn` 是否传入，无则触发 `error` 事件；
2. **触发请求**：触发 `search` 事件，同时更新 `isLoading` 为 true；
3. **请求处理**：
   - 自动中止重复请求（避免并发混乱）；
   - 调用 `requestFn` 传入关键词，接收 Promise 结果；
   - 请求失败触发 `error` 事件，携带错误信息；
4. **结果回调**：请求成功后，触发 `result` 事件，回传后端数据；
5. **状态重置**：请求完成（成功/失败）后，更新 `isLoading` 为 false。

## 五、关键边界场景处理逻辑（框架容错与兼容性）

### 1. 模式配置错误

- 若 `mode` 不在枚举范围内，触发 `error` 事件，提示“无效的搜索模式”；
- 若 `mode="dom"` 但未传入 `targetRef`，触发 `error` 事件，提示“缺少目标组件引用”；
- 若 `mode="remote"` 但未传入 `requestFn`，触发 `error` 事件，提示“缺少远程请求函数”。

### 2. 重复搜索与防抖

- 开启防抖时，输入过程中延迟 `debounce` 时长执行搜索，未到时长则重置计时；
- 远程模式下，新搜索触发时自动中止上一个未完成请求，避免结果覆盖混乱。

### 3. 重置逻辑（全模式通用）

1. 清空输入框内容，重置 `currentKeyword`；
2. 清除文本高亮（仅 `text` 模式）；
3. 调用目标组件 `resetFn` 恢复原始状态（仅 `dom` 模式）；
4. 重置 `resultTotal`、`currentResultIndex`、`isLoading`；
5. 触发 `reset` 事件，通知业务层重置完成。

### 4. 无结果处理

- 任意模式匹配无结果时，触发 `empty` 事件；
- 同步更新 `resultTotal` 为 0，`currentResultIndex` 为 -1；
- 支持业务层自定义“无结果”提示样式，搜索组件不内置 UI。

## 六、目标组件适配规范（dom 模式）

为实现搜索组件与业务组件的高效协作，**所有需参与 DOM 筛选的组件**（菜单、Tree、表格、Select 等）需遵循以下规范：

1. 必须暴露 `filter(keyword: string): Promise<{ total: number, list: 数组 }>` 方法，负责内部匹配逻辑；
2. 必须暴露 `resetFilter(): void` 方法，负责恢复组件原始状态（折叠、显隐、选中）；
3. 匹配逻辑需支持 `scopeSelector`、`excludeSelectors` 配置，限定搜索范围；
4. 支持搜索结果计数回传，与搜索组件 `result` 事件格式对齐。

## 七、最终架构总结

搜索组件以\*\*“统一入口 + 指令分发”\*\*为核心，通过分层设计隔离公共逻辑与业务逻辑，完美适配轻量级 UI 框架需求：

- **dom 模式**：组件自主筛选，不破坏布局与状态；
- **text 模式**：自主处理文本高亮，轻量高效；
- **remote 模式**：统一请求分发，降低业务代码冗余。

未来扩展新搜索场景时，仅需新增模式或适配组件 API，无需重构搜索组件核心逻辑，实现真正的“通用化、框架化”设计。
