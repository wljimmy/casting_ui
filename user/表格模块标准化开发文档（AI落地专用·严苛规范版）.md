# 表格模块标准化开发文档（AI落地专用·严苛规范版）

**文档定位**：纯工程落地强制规范，无模糊描述、无备选方案、仅唯一执行标准，供AI直接1:1编码实现，所有规则强制生效，禁止自定义发挥。

**全局依赖约束**：表格核心数据运算、生命周期管理、渲染逻辑均内聚在 table\.js \+ table\.css 双文件中闭环实现，不引入任何第三方组件、外部插件与外部资源；允许调用框架原生通用模块（如 Input 输入组件、消息提示模块等）作为 UI 交互载体，用于搜索框、操作提示等场景，无需重复开发基础组件。

### 补充说明（文档逻辑一致性校验）

1. 该调整仅放开UI 交互层的框架通用组件复用权限，核心数据层（数据处理、筛选排序、分页、远程请求）仍保持极简原生实现，其中远程请求沿用原生 Fetch API、不依赖框架请求模块，与原有规则无冲突。

2. 初始化模块的框架顶层监听器注册、注册表体系，本身就属于框架原生基础能力复用范畴，与本次调整逻辑完全自洽。

3. 所有复用的框架模块均为框架内置标准能力，不会引入额外外部依赖，双文件核心闭环的架构定位保持不变。

### 1、样式模块（固定不变）

保留当前所有内容，仅允许增量扩展框架原生样式。冻结由CSS完成（放弃不兼容的浏览器），仅负责表格基础UI、冻结样式、功能区样式、分页区样式渲染支撑。**零数据逻辑、零交互逻辑、零状态管理**。

## 2、全局强制分层架构（不可突破）

全局分层隔离为硬性语法规则，所有模块严格遵守，越界即为BUG：

- **数据层模块合集**（数据处理、排序筛选搜索、分页、分页器生成、用户数据更新模块）：**仅读写全局表格注册表**，零DOM操作、零视图渲染、零页面交互，只做数据清洗、规则运算、数据裁切、配置生成、数据更新。

- **渲染层模块**（页面构建模块）：**只读注册表、不修改任何数据与规则**，不参与数据运算，仅监听注册表属性变更，执行DOM热更新与交互事件挂载。所有交互动作**必须调用数据层API**，禁止直接操作注册表。仅对 initStatus=success 的表格生效。

- **初始化模块**：系统唯一表格入口，依托**框架顶层统一注册式DOM监听器**（无组件独立Observer、防冲突、防冗余），负责页面表格扫描、识别、注册、初始化、状态标记、生命周期管控，独立注册表分类，对标Input组件注册表体系。自带原生容错降级机制，严格遵循「功能可失效、数据必展示」的底层兜底原则，保障极端场景下原始数据正常展示。

## 3、全局硬性阈值与限制规则（防内存溢出、防性能雪崩）

所有阈值为强制硬限制，代码层面直接拦截，不兼容超量场景，彻底杜绝大数据、多表格内存溢出问题。

#### 3\.1 前端可处理最大数据量阈值（浏览器性能评估定稿）

基于现代主流浏览器（Chrome/Edge/Firefox）内存、JS执行栈、DOM渲染性能实测评估：

- **前端纯内存运算上限：5000条有效数据行**

- ≤5000条：前端完整支持初始化、筛选、排序、分页、热更新全能力

- \>5000条：前端直接拦截，**拒绝前端数据运算**，强制交由后台分页、筛选、排序处理，前端仅做静态渲染展示

- 限制范围：单表格实例 processedData 基准数据行数，原始导入数据超量直接截断并控制台提示，不报错崩溃

#### 3\.2 筛选/排序规则堆叠上限

- 单表格实例 **filterRules 最大堆叠10条**，超过10条直接拒绝新增规则，控制台输出错误，不生效、不覆盖旧规则

- 单表格实例 **sortRules 最大堆叠10条**，超过10条直接拒绝新增规则，控制台输出错误，不生效、不覆盖旧规则

- 规则清空、替换操作不受上限限制

#### 3\.3 热更新防抖时长（固定配置）

- 全局默认防抖时长：**500ms**

- 支持配置开启长防抖：1000ms，适配超大列表防频繁渲染抖动

#### 3\.4 错误展示强制规范

- **页面零错误UI提示**，所有初始化错误、数据解析错误、规则错误、请求错误、超量错误，**仅浏览器控制台输出日志**

- 页面始终保持稳定渲染状态，不弹窗、不空白、不报错阻断页面

## 4、生命周期与DOM卸载GC机制（定稿闭环）

完全沿用框架原生组件生命周期逻辑，无需手动销毁，自动垃圾回收：

- 表格DOM节点卸载、销毁、页面移除时，框架顶层监听器捕获节点销毁事件

- 自动匹配当前DOM节点对应的注册表 tableId，**从全局注册表中彻底删除该表格实例对象**

- 实例对象销毁后，所有关联监听、规则、数据、分页状态同步失效，JS引擎自动GC回收内存

- 无内存残留、无无效监听、无僵尸注册表数据，完美适配多表格动态新增/删除场景

**结论**：无需额外编写销毁逻辑，框架原生注册表绑定DOM生命周期，DOM销毁 = 注册表对象销毁 = 内存释放，逻辑自洽。

## 5、远程数据源强制规范

- 远程数据请求**不依赖任何框架模块**，底层直接使用原生 Fetch API 实现，极简无依赖

- 仅支持两种远程文件格式：**纯JSON、纯CSV**，其他格式（xls/xlsx/txt等）直接拒绝拉取

- 拉取资源格式不符合规范、文件损坏、数据格式非法、层级错乱时，直接终止初始化，表格置为 error 状态

- **前端零纠错逻辑**：不做数据兼容、不做格式修复、不做容错补全，不符合规范直接拒绝，控制台打印异常信息

- 远程数据源拉取失败、超时、跨域，统一置为初始化 error 状态

## 6、注册表结构升级（新增持久化、补齐缺失字段）

补齐历史缺失字段，实现**筛选、搜索、排序、分页全状态注册表持久化**，彻底解决渲染模块手动传参问题，渲染模块只读注册表状态即可。

新增字段：searchRules（搜索规则持久化）、pageState（分页状态持久化）

```json
{
  "tableId": "table_001",
  "header": ["id", "name", "age", "department"],
  // 四状态机：pending/loading/success/error
  "initStatus": "success",
  "initError": "",
  // 数据变更时间戳，全局统一更新
  "updateTime": 1718960000000,
  // 原始只读数据（仅初始化/用户更新模块可写入）
  "rawData": [],
  // 基准固化数据（前端运算唯一数据源）
  "processedData": [],
  // 筛选排序后结果数据
  "filteredData": [],
  // 筛选规则队列（最大10条）
  "filterRules": [],
  // 排序规则队列（最大10条）
  "sortRules": [],
  // 新增：搜索规则持久化（补齐历史缺失）
  "searchRules": {
    "keyword": "",
    "mode": "fuzzy", // fuzzy/exact/regex
    "field": "all" // all / 指定字段名
  },
  // 新增：分页状态全局持久化（彻底无需手动传参）
  "pageState": {
    "pageNum": 1,
    "pageSize": 50,
    "total": 0,
    "pageCount": 0
  }
}
```

## 7、渲染模块精细化热更新策略（核心重大优化）

严格区分**局部热更新**与**全量重建**，最小化DOM操作，极致优化性能，杜绝不必要重渲染。

#### 7\.1 局部热更新（90%场景）

触发条件：**表头不变、数据源结构不变**，仅以下内容变更：数据内容、筛选规则、排序规则、搜索关键词、分页切换

执行规则：

- 不重新渲染顶部功能区、不重建表头thead、不重建表尾tfoot、不修改分页栏结构

- **仅替换表格tbody数据区域DOM**，仅更新单元格内容

- 保留所有冻结状态、UI状态、按钮状态、滚动位置

- 防抖500ms合并更新，避免频繁DOM操作

#### 7\.2 全量重建表格（重大变更场景）

触发条件（满足任意一条即全量重建）：

- 表头header字段新增/删除/修改/顺序变更

- 整体数据源替换、数据结构层级变更

- 用户新推送数据与原表头不匹配

执行规则：

- 销毁当前完整表格DOM结构，重建整体表格

- 清空所有历史规则：filterRules、sortRules、searchRules

- 重置分页pageState为初始状态（pageNum:1,pageSize:50）

- 重新执行数据标准化、注册表更新、全量渲染

- 同步更新updateTime时间戳

## 8、全模块独立详细开发说明（完整恢复，AI开发核心依据）

本章节完整恢复所有表格核心模块的**独立定位、触发时机、使用方法、返回值、内部逻辑、约束规则**，为AI逐模块开发提供完整依据，无缺失、无精简。

### 8\.1 初始化模块（核心前置模块）

**模块定位**：系统前置自动执行核心模块，依托框架顶层统一注册式DOM监听器全局常驻托管，是所有表格功能的唯一入口，负责页面表格扫描、类型区分、注册表注册、数据初始化、状态标记、容错降级。自带原生防错兜底机制，保证功能崩溃不影响原始数据展示。

**触发时机**：框架DOM加载完成后全局自动执行，监听器常驻后台，动态监听页面表格节点变更

**使用方式**：全自动执行，无手动调用入口，由table\.js内部统一调度

**核心能力**：区分普通表格/数据驱动表格、四状态机管控、容错降级、防重复初始化、异常节点跳过

#### 8\.1\.1 节点扫描跳过规则（最终正确版，彻底防重复初始化）

全局扫描未初始化CUI\-Table节点时，**分层判定、强制跳过两类节点**，从机制上彻底杜绝重复初始化、无效重试、冗余执行，保证单次节点生命周期内初始化逻辑唯一执行：

- **1\. 永久跳过：初始化完成节点（DOM持久标记）**：带有 `Data-table-init=finish` DOM属性的表格，代表已完整初始化成功，永久跳过，适配页面刷新场景

- **2\. 运行跳过：初始化异常节点（注册表实例状态）**：已存在对应注册表实例，且 `initStatus=error` 的表格，无需操作DOM、无需二次校验节点，直接内存判定跳过，杜绝重复重试初始化

#### 8\.1\.2 初始化容错降级规则（架构自洽定稿）

- **初始化成功**：删除原生Table内部冗余DOM结构，仅保留外层容器，DOM追加 `Data-table-init=finish` 持久标记，注册表实例置为 success 状态，开启全量功能与热更新

- **初始化失败**：**全程不操作任何DOM属性、不新增任何DOM标记**，仅将当前表格对应的注册表实例 `initStatus` 置为 error 状态；保留完整原始Table DOM结构用于静态展示，CSS样式自动生效，仅高级功能失效，控制台输出错误日志，后续扫描直接通过注册表状态跳过该节点，不再重复尝试初始化

#### 8\.1\.3 标准化初始化执行时序（最终定稿）

1. 框架加载完成，向框架顶层监听器注册表格专属监听事件，统一托管，杜绝监听冲突与冗余

2. 全局扫描页面所有 \.CUI\-Table 节点，依据双向判定规则自动跳过两类节点：DOM判定已初始化完成节点、注册表判定初始化错误节点，仅处理未初始化pending节点

3. 区分表格类型：普通表格仅加载CSS样式，无需注册、无需初始化；数据驱动表格继续执行后续流程

4. 数据驱动表格优先创建注册表空实例，初始状态默认为 pending

5. 实例创建完成，即刻修改 initStatus 为 loading 加锁，禁止所有监听与重复初始化

6. 解析数据源（静态Table/JSON/CSV/远程数据源），远程数据源仅放行JSON、CSV格式，非法格式直接判定初始化失败

7. 调用数据处理模块完成数据清洗、标准化，强制校验5000条数据上限，超量拦截前端运算

8. **初始化成功分支**：挂载rawData、processedData、filteredData三层数据，初始化规则队列、分页状态、更新时间戳，修改状态为success解锁，清理原生冗余DOM，添加初始化完成标记

9. **初始化失败分支**：挂载错误信息，仅修改注册表实例状态为error，**不操作DOM、不标记DOM异常属性**，保留原始完整Table DOM结构，仅禁用所有高级功能，CSS样式自动生效，后续扫描自动跳过该异常实例

### 8\.2 数据处理模块（基础数据核心模块）

**模块定位**：纯数据层基础模块，无任何DOM操作、无渲染逻辑，负责将各类非标数据源统一清洗为标准键值对表格数据，生成可热更新的注册表表格实例，是所有前端表格运算的唯一数据来源。

**触发时机**：初始化模块自动调用、用户数据更新模块主动调用

**使用方法**：CUI\.Table\.xxx（预设表头，待清理数据）

**返回值**：统一返回 \{code:状态码, tableId:表格唯一标识, msg:提示信息\}

**核心能力**：多格式数据兼容、数据清洗、表头适配、空值补全、无效数据剔除、标准化结构输出、超量数据拦截

**执行规则**：仅读写注册表数据，不操作DOM，不参与视图渲染，不修改原始数据源

### 8\.3 排序筛选搜索模块（数据运算模块）

**模块定位**：纯数据层运算模块，基于注册表基准数据完成检索、多条件筛选、多条件排序，持久化所有规则，输出过滤后的标准数据集，供分页、渲染模块使用。

**触发时机**：用户交互触发、注册表规则变更触发、数据更新后自动重算

**使用方法**：CUI\.Table\.xxx\(tableId, 操作参数\)

**返回值**：\{code:状态码, total:过滤后总条数, filterRules:生效筛选规则, sortRules:生效排序规则, msg:提示信息\}

**核心约束**：

- 规则堆叠模式：新规则追加至队列末尾，最大堆叠10条，超量拦截

- 重算机制：所有规则变更均基于processedData基准数据全量重算，无增量叠加误差

- 固定执行顺序：关键词检索 → 顺序筛选 → 顺序排序

- 纯数据运算，不修改原始数据，不操作DOM

### 8\.4 分页模块（数据裁切模块）

**模块定位**：纯数据层裁切模块，职责单一，仅负责对过滤后的全量数据进行分页裁切，输出当前页标准数据，不参与筛选、排序、数据清洗。

**触发时机**：渲染模块热更新、用户切换页码/每页条数、数据重算后自动调用

**使用方法**：CUI\.Table\.xxx\(tableId, 当前页码, 每页条数\)

**返回值**：\{code:状态码, data:\{header,rows\}, pageInfo:分页统计信息, msg:提示信息\}

**核心约束**：参数容错兜底、边界值处理、空数据兼容、只读裁切不修改注册表原始数据

### 8\.5 分页器生成模块（配置生成模块）

**模块定位**：数据层内部依赖模块，仅供给渲染模块调用，不对外暴露业务入口，负责基于注册表分页状态生成完整分页UI配置。

**触发时机**：渲染模块每次热更新、全量重建时内部自动调用

**使用方法**：内部私有调用，无外部API

**返回值**：\{code:状态码, pagerConfig:分页完整配置, msg:提示信息\}

**核心能力**：页码生成、按钮状态判定、边界容错、适配UI渲染

### 8\.6 页面构建渲染模块（唯一视图模块）

**模块定位**：系统唯一DOM渲染模块，纯视图层，只读注册表数据，不参与任何数据处理，仅监听success状态表格的注册表变更，实现局部热更新与全量重建。

**触发时机**：初始化成功后首次全量渲染、注册表数据/规则变更防抖触发、用户数据更新触发

**使用方法**：CUI\.Table\.xxx\(tableId, 渲染容器, 渲染配置\)

**返回值**：渲染实例对象（含手动刷新、销毁、页码切换方法）

**核心强制约束**：仅监听 \& 渲染 initStatus=success 的表格，其余状态完全静默，不执行任何渲染逻辑

**渲染结构**：统一渲染顶部功能区、中间冻结表格区、底部状态分页区

### 8\.7 用户数据对接更新模块（对外开放API模块）

**模块定位**：对外唯一数据更新入口，纯数据层模块，开放给业务用户调用，支持同页面无刷新动态替换表格原始数据，适配业务实时更新场景。

**触发时机**：用户业务代码主动调用

**使用方法**：CUI\.Table\.updateData\(tableId, newOriginData\)

**返回值**：统一状态结果，由渲染模块自动适配更新视图

**核心逻辑**：表头一致仅局部更新数据，表头不一致全量重建表格，超量数据拦截前端运算

## 9、筛选排序规则堆叠逻辑定稿

- 所有追加规则为**纯堆叠模式**，新规则追加至规则队列末尾

- 每次规则变更（新增/删除/清空），**基于原始 processedData 基准数据全量重算**，不增量叠加，杜绝数据累积误差

- 严格遵循执行顺序：检索 → 筛选（队列顺序） → 排序（队列顺序）

- 超10条规则直接拦截，不生效、不覆盖、不报错崩溃

## 10、全模块BUG修复与逻辑统一（全部定稿）

#### 10\.1 修复渲染层越界问题
